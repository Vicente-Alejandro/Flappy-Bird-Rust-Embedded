# Migration Guide: Bevy 0.18 → 0.19

**Status:** ✅ COMPLETED  
**Date:** 2026-07-20  
**Rust toolchain:** `stable 1.97.1` (x86_64-pc-windows-msvc)  
**System:** Windows 11 Home — Intel i5-10300H — GTX 1650 — 16 GB RAM  

---

## Executive Summary

The migration from Bevy 0.18.1 → 0.19.0 **required zero changes to application source code** (`src/main.rs`). All work was focused on resolving a **transitive dependency bug** in the Bevy 0.19.0 ecosystem caused by incompatible versions of the `windows` crate in `wgpu`'s DX12 backend.

---

## The Problem

### Error Description

Running `cargo run` with `bevy = "0.19.0"` produced ~10 compiler errors inside the external crate `wgpu-hal-29.0.4/src/dx12/suballocation.rs`:

```
error[E0308]: mismatched types
  --> wgpu-hal-29.0.4\src\dx12\suballocation.rs:83:71
   |
   | device: gpu_allocator::d3d12::ID3D12DeviceVersion::Device(raw.clone()),
   |         expected `windows::Win32::Graphics::Direct3D12::ID3D12Device`
   |         found `Direct3D12::ID3D12Device`

error[E0277]: the trait bound `ResourceCategory: From<&D3D12_RESOURCE_DESC>` is not satisfied
error[E0277]: the trait bound `&ID3D12Heap: Param<ID3D12Heap, InterfaceType>` is not satisfied
```

The errors were **not in our code** — they were deep inside Bevy's dependency graph.

### Root Cause: `windows` Crate Version Diamond Conflict

The problem was a **diamond dependency conflict** on the Microsoft `windows` crate:

| Crate | Required `windows` version | Resolved version |
|---|---|---|
| `wgpu-hal 29.0.4` | `= "0.62"` | `0.62.2` ✅ |
| `gpu-allocator 0.28.0` | `">=0.53, <=0.62"` | `0.58.0` ❌ |
| `cpal 0.17.3` (audio) | `">=0.59, <=0.62"` | `0.58.0` |
| `gilrs 0.11.0` (gamepads) | `"0.58"` | `0.58.0` |

`gilrs` (Bevy's gamepad input crate) pinned `windows 0.58.0` in the dependency graph. When Cargo resolved `gpu-allocator` (which accepts `>=0.53, <=0.62`), it picked `0.58.0` because it was already present — instead of `0.62.2`.

**Consequence:** `wgpu-hal 29.0.4` compiled its DX12 types against `windows 0.62.2`, but `gpu-allocator 0.28.0` compiled its DX12 types against `windows 0.58.0`. In Rust, these are **two completely separate, incompatible instances** of the same types (`ID3D12Device`, `ID3D12Heap`, `D3D12_RESOURCE_DESC`) even though they share the same name.

## Diagnosis

### Diagnostic Commands Used

```powershell
# Show all versions of the `windows` crate in the dependency graph
cargo tree -p windows
# Output revealed three concurrent versions: 0.58.0, 0.61.3, 0.62.2

# Reverse dependency: who depends on windows 0.61.3?
cargo tree -i "windows@0.61.3"
# Output: gpu-allocator <- wgpu-hal <- wgpu <- bevy_render <- bevy

# Reverse dependency: who depends on windows 0.58.0?
cargo tree -i "windows@0.58.0"
# Output: gilrs-core <- gilrs <- bevy_gilrs <- bevy_internal <- bevy
```

---

## Failed Attempts

### Attempt 1: `cargo update windows@0.61.3 --precise 0.62.2`

**Result:** Partially successful but insufficient.

The command removed `windows 0.61.3` from the lock file, but Cargo then resolved `gpu-allocator` to `windows 0.58.0` (already present via `gilrs`). The error changed from "0.61 vs 0.62" to "0.58 vs 0.62" — still broken.

---

## Final Solution: Local Patch of `gpu-allocator`

### Strategy

Use Cargo's `[patch.crates-io]` feature to replace `gpu-allocator 0.28.0` with a local copy that pins `windows` explicitly to `"0.62"`, eliminating the version ambiguity during resolution.

### Step 1: Copy the crate into the project

```powershell
Copy-Item -Recurse -Force `
  "C:\Users\<user>\.cargo\registry\src\index.crates.io-<hash>\gpu-allocator-0.28.0" `
  "patches\gpu-allocator"
```

### Step 2: Modify `patches/gpu-allocator/Cargo.toml`

Change the `windows` dependency from a loose range to an exact version:

```toml
# BEFORE (original Cargo.toml.orig):
[target.'cfg(windows)'.dependencies.windows]
version = ">=0.53, <=0.62"

# AFTER (patched):
[target.'cfg(windows)'.dependencies.windows]
version = "0.62"
```

Also update `dev-dependencies`:
```toml
# BEFORE:
[target.'cfg(windows)'.dev-dependencies.windows]
version = ">=0.58, <=0.62"

# AFTER:
[target.'cfg(windows)'.dev-dependencies.windows]
version = "0.62"
```

### Step 3: Register the patch in the project's `Cargo.toml`

```toml
# Fix: wgpu-hal 29.x uses windows 0.62 DX12 types, but gpu-allocator 0.28.0
# resolves to windows 0.58.0 (via gilrs), causing type mismatches in
# suballocation.rs. This patch forces gpu-allocator to use windows 0.62.
[patch.crates-io]
gpu-allocator = { path = "patches/gpu-allocator" }
```

### Verification

After applying the patch, `cargo build` completed without errors:

```
Compiling gpu-allocator v0.28.0 (patches/gpu-allocator)  ← local patch active
Compiling wgpu-hal v29.0.4                                ← no errors
Compiling wgpu v29.0.4
Compiling bevy_render v0.19.0
...
Finished `dev` profile [unoptimized + debuginfo]
```

## Runtime Notes

### Non-critical warnings on startup

These messages appear at runtime and are **safe to ignore**:

```
ERROR wgpu_hal::vulkan::instance: GENERAL [Loader Message (0x0)]
    loader_get_json: Failed to open JSON file
    C:\Program Files (x86)\Overwolf\0.294.3.2\ow-vulkan-overlay64.json
```

**Cause:** Overwolf (gaming overlay) registers Vulkan layer extensions on the system, but the JSON files are outdated or missing. Does not affect application behavior.

Bevy 0.19 auto-selects the best available backend. On this system it chose **Vulkan** over DX12:

## Bevy 0.19 Notable Breaking Changes

These did not affect this project, but are relevant for future reference:

- **Render Graph → Systems**: The render graph now runs as normal ECS systems. Custom render passes require significant changes.
- **`bevy_material`**: New separate crate for materials.
- **`bevy_scene_macros`**: New scene macro crate.
- **`bevy_clipboard`**: New clipboard crate.
- **`bevy_world_serialization`**: New world serialization crate.
- **`Query::single()`**: Now returns `Result` — use `.unwrap()` or `if let Ok(...)`. *(Our code was already compatible.)*

See the [official Bevy 0.18 → 0.19 migration guide](https://bevyengine.org/learn/migration-guides/0-18-to-0-19/) for the full list.

## Reproducibility in Other Environments

For another developer to compile this project from scratch:

1. **Clone the repo** — the `patches/` directory is committed to git
2. **Rust toolchain:** `stable` ≥ 1.87.0 (required by wgpu 29.x). Recommended: `1.97.1`
3. **Windows:** MSVC toolchain required (`x86_64-pc-windows-msvc`), not GNU
4. **Build Tools:** Visual Studio Build Tools with "Desktop development with C++" installed
5. Run `cargo run` — the patch is applied automatically by Cargo

> [!IMPORTANT]
> The `patches/gpu-allocator/` directory **must be present in git**.  
> Without it, the `[patch.crates-io]` entry fails and the project will not compile on Windows with DX12 enabled.
